2. 项目定位与架构

这是一个“本地桌面软件 + 浏览器界面 + SQLite 本地数据库”的快递运费结算系统。核心目标是：批量导入 Excel、按规则计算运费、输出结果并保存历史。

2.1 技术栈

层技术说明
前端Vue 3 + TypeScript + Vite + Pinia + Element Plus负责界面、路由、状态管理
后端Go提供 HTTP API、业务逻辑、数据库访问
数据层SQLite本地存储规则、历史、授权信息
Excelexcelize/v2读取/导出 Excel
打包Go + PowerShell / Batch生成 Windows 可执行文件

2.2 核心目录

main.go                      # 程序入口、HTTP 路由、前端服务
internal/app/app.go         # 应用门面层，统一业务调用
internal/db/sqlite.go       # 数据库初始化与迁移
internal/freight/           # 运费计算引擎
internal/rules/             # 规则模型、仓储、匹配逻辑
internal/license/           # 授权系统
frontend/src/               # Vue 前端页面与状态管理
server_php/                 # PHP 授权服务端（可选）

3. 后端开发指南

3.1 入口文件

主程序入口在 main.go，职责包括：

  • 启动 HTTP 服务
  • 注册所有 API 路由
  • 嵌入前端静态资源
  • 处理授权与登录
  • 处理 Excel 上传/预览/计算/导出

3.2 常用模块职责

文件职责
internal/app/app.go统一暴露业务能力，适合作为调用总入口
internal/db/sqlite.go数据库连接、schema 初始化、写锁控制
internal/freight/engine.go基础运费、附加费用、全局加价计算
internal/freight/avgweight.go拉均重 / 偏差加价
internal/rules/repository.go规则查找、规则索引、活动时间判定
internal/rules/models.go数据结构定义
internal/rules/brackets.go区间计费逻辑
internal/license/授权与机器码相关逻辑

3.3 后端开发规范

  • 数据库写入优先使用 db.WriteExec()
  • 避免直接裸写 SQLite，防止锁竞争
  • 新增接口时，优先复用现有 decodeJSONBody 之类的解析方式
  • 大批量计算要尽量走规则索引和预加载逻辑
  • 改动计费逻辑时，一定要补测试

3.4 常见后端新增功能流程

1
在 internal/ 下新增对应模块或复用现有模块
2
在 main.go 中注册新的 HTTP 路由
3
在前端 frontend/src/api/index.ts 中补接口封装
4
在前端页面中接入调用，更新路由或状态管理
5
执行 go test ./... 与前端构建验证

4. 前端开发指南

4.1 页面结构

frontend/src/views/
  Layout.vue      # 布局，左侧导航
  Home.vue        # 首页
  Calc.vue        # 计算页面
  Rules.vue       # 规则管理
  History.vue     # 计算历史
  Settings.vue    # 设置
  License.vue     # 授权
  Login.vue       # 登录
  TestRule.vue    # 规则测试

4.2 状态管理

当前使用 Pinia，状态入口在 frontend/src/stores/app.ts。

  • 页面间共享的数据建议放到 Pinia
  • 组件内部只保留局部状态
  • 接口调用尽量封装到 frontend/src/api/index.ts

4.3 路由与新增页面

在 frontend/src/router/index.ts 中注册新页面。

新增页面一般要做三件事：

  1. 在 src/views/ 下新增 Vue 文件
  2. 在 src/router/index.ts 注册路径
  3. 在 src/views/Layout.vue 的导航菜单里补入口

4.4 前端开发约定

  • 优先使用 Composition API
  • 所有接口请求尽量统一封装，不要散落在组件中
  • 页面交互复杂时，先抽象成 composable 或 store
  • 打包前确保 npm run build 无错误

5. 规则与计费体系

5.1 规则类型

类型用途优先级
campaign活动规则，带时间范围最高
customer客户规则较高
global全局规则中等
default默认规则最低

5.2 匹配顺序

campaign → customer → global → default

同级别里，精确省份优先于全国通配（空省份）。

5.3 计费模式

  • simple：首重续重
  • bracket：区间计费，支持一口价和首续重混合

5.4 续重单位

  • full_kg：按整公斤向上取整
  • hundred_gram：按 100g 为单位
  • actual_weight：按实际重量

5.5 额外费用叠加顺序

基础运费
+ surcharge
+ 省份附加费
+ 最低/最高价限制
+ 全局加价
= 最终运费

5.6 拉均重规则

拉均重规则主要在 internal/freight/avgweight.go 中实现，流程是：

按客户分组统计重量
→ 过滤超上限包裹
→ 计算平均重量
→ 与基准重量比较
→ 按 step_weight / step_price 计算加价
→ 分摊到每件订单

6. 数据库说明

项目用 SQLite，本地文件存储，数据文件通常位于用户目录下，Windows 下多半是 %APPDATA%/yunfei/yunfei.db。

6.1 重要表

表名作用
freight_rules运费规则
freight_weight_brackets区间计费规则数据
global_rules全局保底与加价配置
global_province_surcharges省份附加费
calc_history历史计算记录
license_info授权信息

6.2 开发注意事项

  • 数据库修改优先通过现有仓储函数
  • 写入请走 db.WriteExec
  • 避免长时间占用数据库连接
  • 如果你改了表结构，要同时更新初始化逻辑和迁移逻辑

7. 主要 API 与调用方式

后端是典型的 HTTP API 层，前端通过 frontend/src/api/index.ts 调用。

接口用途
/api/rules获取规则列表
/api/rules/save保存规则
/api/rules/test测试规则命中结果
/api/global-rules获取/设置全局规则
/api/province-surcharges省份附加费管理
/api/avg-weight-rules拉均重规则管理
/api/calculate单次计算
/api/calculate/batch批量计算
/api/excel/upload上传 Excel
/api/excel/previewExcel 预览
/api/history历史记录

新增 API 时，建议遵循这种模式：

在 main.go 中注册路由
→ 在 internal/app/app.go 中封装业务方法
→ 在 frontend/src/api/index.ts 中暴露前端调用

8. 如何继续开发

8.1 如果你要新增一个“规则类型”

  1. 先在 internal/rules/models.go 扩展结构体
  2. 在 internal/rules/repository.go 中完善保存/读取逻辑
  3. 在 internal/freight/engine.go 中让计费逻辑识别新类型
  4. 在前端规则管理页面补字段
  5. 补测试并验证

8.2 如果你要新增一个页面

  1. 在 frontend/src/views/ 新建 Vue 文件
  2. 在 frontend/src/router/index.ts 注册路由
  3. 在 frontend/src/views/Layout.vue 增加菜单入口
  4. 如果需要数据，则在 frontend/src/api/index.ts 和后端接口中加

8.3 如果你要新增一个后端接口

1. 在 main.go 中注册路由
2. 在 internal/app/app.go 中写业务方法
3. 由前端通过 api/index.ts 调用
4. 记得做参数校验与错误处理

8.4 如果你要修复一个 BUG

  • 先复现问题，再定位根因
  • 尽量只改一个点，避免一锅端
  • 如果是计费逻辑，优先写回归测试
  • 修完后运行 go test ./... 和前端构建

当前项目提供了构建脚本：

  • build.ps1

若打包遇到编码或文件名问题，优先检查脚本里的输出名和 PowerShell 的字符编码。

10. 常见问题

现象可能原因处理方式
前端无法访问后端接口后端未启动或端口不对确认 go run . 已启动，且 Vite 代理指向 58080
数据库被锁写入并发过多使用 db.WriteExec，避免直接并发写
规则没生效规则未启用、日期不在范围、客户名/省份不匹配检查规则状态、时间字段、Normalize 逻辑
打包后的 exe 名称异常PowerShell 字符编码或脚本输出名问题检查构建脚本中输出文件名，重新生成

10.1 推荐开发顺序

  1. 先跑通前后端
  2. 定位你要改的模块
  3. 先看现有相似功能怎么做
  4. 小步修改、频繁验证
  5. 最后做打包确认
